## 遇到的问题和解决
1. 问题：非 root 用户启动 nginx 时 `/var/run/nginx.pid` 无写权限。
   解决：在 Dockerfile 中预创建 `nginx.pid` 并 `chown appuser`，同时把缓存目录 `/var/cache/nginx`、日志目录 `/var/log/nginx` 都 chown 给 `appgroup`，避免运行时权限报错。

2. 问题：SPA 路由刷新页面后 404（直接访问 `/task/1` 等子路由）。
   解决：在 `nginx.conf` 中加入 `try_files $uri $uri/ /index.html;`，让所有未命中静态资源的请求 fallback 到 `index.html`，由 React Router 接管路由。

3. 问题：生产环境前后端跨域 / API 地址硬编码问题。
   解决：前端构建后通过 nginx 的 `/api/` location 反向代理到 `http://backend:8080/api/`，前端代码统一调用相对路径，无需关心后端实际地址。

## 心得体会
完成 Docker 化最大的收获是理解了"构建环境"与"运行环境"分离带来的好处：用 node 镜像编译出静态资源，再用一个仅含 nginx 的 alpine 镜像分发，最终前端镜像可压缩到 50MB 以内，并且运行时不带任何 npm / 源码，攻击面显著缩小。

另一点是 Docker Compose 的健康检查 + `depends_on.condition: service_healthy` 组合，让前端能够确保 backend 已经真正可用再启动，避免了启动顺序竞争问题，这比之前用 `sleep` 等待更优雅可靠。

最后，把 nginx 的反向代理写进前端镜像本身，让前后端在同一个网络命名空间下通信，开发与生产环境的 API 调用方式可以完全一致，省去了大量跨域配置工作。
