### 3. 问题解决

#### 问题 1：Monorepo 结构下 Vercel 构建路径

- **现象**：项目根目录是 `HelpMate/`，前端代码在 `frontend/` 子目录，Vercel 默认在根目录执行 `npm run build` 会导致找不到 `package.json`
- **解决**：在 `vercel.json` 中设置 `"rootDirectory": "frontend"` 和 `"buildCommand": "npm run build"`，Vercel 会 `cd frontend` 后执行构建命令

#### 问题 2：SPA 路由刷新 404

- **现象**：直接访问 `/task/1` 等前端子路由时，Vercel 尝试匹配静态文件，找不到返回 404
- **解决**：配置 rewrites 规则 `{"source": "/((?!api/).*)", "destination": "/index.html"}`，将非 API 路径全部 fallback 到 `index.html`，由 React Router 接管路由分发。排除 `/api/` 路径避免影响后端代理。

#### 问题 3：前端如何知道后端 API 地址

- **现象**：本地开发 API 在 `localhost:8080`，云部署时后端地址完全不同
- **解决**：通过 `VITE_API_BASE_URL` 环境变量配置。本地 `.env` 中设置为 `http://localhost:8080/api`，Vercel 生产环境中设置为线上后端地址。前端 `api.js` 统一通过 `import.meta.env.VITE_API_BASE_URL` 读取，无需修改代码。

#### 问题 4：Vercel 上传包含无用文件

- **现象**：`node_modules`、`backend`、`docs` 等文件被上传到 Vercel，浪费部署时间和空间
- **解决**：创建 `.vercelignore` 文件，排除 `node_modules`、`backend`、`docs`、`.git`、`coverage`、测试文件等

## 心得体会

这次云部署让我对"前端部署的完整链路"有了更清晰的理解。之前觉得部署就是把文件丢到服务器上，实际操作后发现有很多细节需要考虑：

1. **环境差异**：本地开发的 `localhost:8080` 到线上完全不同，需要环境变量来桥接。Vite 的 `VITE_` 前缀机制很巧妙，避免了把敏感变量泄露到前端。

2. **SPA 的部署特殊性**：多页面应用每个路由对应一个 HTML 文件，但 SPA 只有一个 `index.html`。如果不配置 URL rewrites，直接访问子路由会 404，这是一个经典的部署陷阱。

3. **部署即代码**：通过 `vercel.json` 把部署配置纳入 Git 版本管理，团队成员可以清楚地看到部署逻辑，新成员上手也更快。

4. **平台选型很重要**：Vercel 对前端项目非常友好，对比自己用 VPS + Nginx + Docker 部署，节省了大量运维工作。但对于后端，Docker Compose 仍然是更灵活的选择，两者可以很好地配合。

总的来说，这次作业让我从"只写前端代码"扩展到了"让前端代码在生产环境跑起来"，对完整的开发-部署闭环有了更深的理解。
