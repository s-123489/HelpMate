## 遇到的问题和解决

1. 问题：浏览器环境无法使用 Winston（Winston 依赖 Node.js `fs` 模块）
   解决：自定义轻量级 logger，参考 Winston 的 API 设计，使用 `console.log`/`console.warn`/`console.error` 输出。生产环境输出 JSON.stringify，开发环境使用 `%c` CSS 样式美化输出，体验不输 Winston。

2. 问题：SPA 应用中如何暴露 `/health` 端点
   解决：前端静态资源由 Nginx 托管，直接在 `nginx.conf` 中用 `location /health` 返回 JSON，不经过 React 应用路由。同时提供前端主动探活后端的 `checkBackendHealth()` 函数，5 秒超时避免长时间等待。

3. 问题：API 服务已有业务逻辑，如何无侵入地添加指标和日志
   解决：在 `request()` 函数开头和结尾插入 `metrics.recordApiCall()` 和 `logger.apiRequest()`，使用 `try/catch` 同时处理成功和失败两种情况。不修改原有业务逻辑，对调用方完全透明。

4. 问题：PerformanceObserver 在某些浏览器（如旧版 Safari）不支持
   解决：每个 Observer 都包裹在 `try/catch` 中，浏览器不支持时静默忽略，不影响主功能。

## 心得体会

这次监控配置作业让我对"前端可观测性"有了更深入的理解。之前觉得监控是后端的事情，前端只需要关注 UI，但实际上：

1. **前端也需要日志**：用户浏览器中发生的错误、API 调用失败、性能瓶颈，这些都是后端无法感知的。结构化日志让调试不再依赖"用户截图"，可以直接通过日志还原问题现场。

2. **健康检查不仅是后端的事**：前端的健康检查可以从用户视角验证"整个系统是否可用"——前端自己是否正常加载、能否连接到后端 API。这是后端单独的健康检查无法覆盖的。

3. **指标的粒度很重要**：不仅是"有没有错误"，而是"哪个 API 慢"、"哪种错误最多"、"哪类用户操作触发问题"。metrics.js 中按 URL、方法、错误类型分类统计，能快速定位问题。

4. **错误边界是 React 应用的安全网**：之前没有 ErrorBoundary 时，一个组件报错可能导致整个页面白屏。有了错误边界，即使局部组件崩溃，其余功能仍可正常使用，用户体验显著提升。

5. **监控要做得轻量**：浏览器环境资源有限，监控代码不能影响应用性能。所有 observer 都是异步的，日志输出也尽量少做字符串拼接，把性能开销降到最低。
